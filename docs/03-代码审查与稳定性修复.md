## 3. 修复 1：chaoxing 连接失败（核心 bug）

### 问题

`chaoxing.py:87` — `httpx.ConnectError: All connection attempts failed`

`httpx` 发 POST 到 `https://passport2.chaoxing.com/fanyalogin` 时 TCP 握手失败。

### 根因

原代码只尝试一次连接，无重试机制。国内校园网环境偶发 DNS 延迟或路由抖动，单次连接失败就直接放弃。

### 修复内容

`agent_service/tools/chaoxing.py` — 以下改动：

1. **连接重试**：`_login_with_retry()` 最多重试 3 次，指数退避（1.5s → 3s → 6s）
2. **代理支持**：新增 `CHAOXING_PROXY` 环境变量，非空时创建带代理的 `HTTPTransport`
3. **超时拆分**：`timeout=15.0` → `httpx.Timeout(15.0, connect=10.0)`（连接和读取分开超时）
4. **诊断函数**：`_print_diagnostic_hints()` 识别 DNS 失败、连接拒绝、超时、SSL 错误、全局连接失败五种场景，打印中文建议

### 验证

```python
# 直接测试，成功抓到 8 门课程
from agent_service.tools.chaoxing import fetch_chaoxing_tasks
print(asyncio.run(fetch_chaoxing_tasks()))
# → 本学期课程（共 8 门）：体育(四)、形势与政策(二)、信号与系统、幸福心理学、
#    数字逻辑、Java程序设计、Python数据分析、数据库系统概论
```

## 4. 修复 2：LLM 调用连接失败

### 问题

`POST /generate-plan` 返回 `LLM 调用失败：Connection error.`

但直接 `httpx.AsyncClient` 测试 DeepSeek API 返回 200，说明网络是通的。

### 根因

`openai` SDK 的 `AsyncOpenAI` 内部创建自己的 `httpx.AsyncClient`，不经过用户的自定义代理配置。系统级代理对 Python 子进程不生效。

### 修复内容

`agent_service/agents/planner.py`：

1. 新增 `_build_http_client()` 构建带代理的 `httpx.AsyncClient`
2. 读取 `LLM_PROXY` 环境变量，非空时注入代理（`proxy` + `verify=False`）
3. 超时上调至 `connect=15s, total=60s`（LLM 生成响应需要更长时间）
4. 将自定义 client 传入 `AsyncOpenAI(http_client=...)`
5. 调用结束后 `await http_client.aclose()` 释放连接

### 使用方式

如需代理，在 `.env` 中：
```ini
LLM_PROXY=http://127.0.0.1:7890
```

## 6. 修复 4：Java RestTemplate 超时配置

### 问题

`RestTemplateConfig.java` 创建 `RestTemplate` 时没设超时。CLAUDE.md 明确要求"超时设 10s"。JDK 默认 `connectTimeout=0`（无限等待），如果 Python 服务挂了，Java 线程会永久阻塞。

### 修复过程

第一次设 `readTimeout=10s`，但生成规划时 LLM + chaoxing 抓取需要 10-15s，导致 Java 侧 `Read timed out` 降级。调整为 `readTimeout=45s` 覆盖正常耗时。

```java
factory.setConnectTimeout(5_000);   // 建立连接超时 5 秒
factory.setReadTimeout(45_000);    // 等待响应超时 45 秒
```

## 8. 修复 6：PlanRequest 字段缺失 + null 值 422

### 第一轮：字段缺失

Java `PlanRequest.java` 没有 `force_refresh` 字段，但 Python Pydantic 模型有。Java 无法主动要求 Python 跳过缓存。

**修复**：Java 端补上 `@JsonProperty("force_refresh") private Boolean forceRefresh;`

### 第二轮：null 值导致 Pydantic 校验失败

补上字段后，Java 端 builder 不给 `forceRefresh` 赋值时，Jackson 序列化为 `"force_refresh": null`。Python 的 `bool` 类型不接 `null`，FastAPI 返回 422：

```json
{"detail":[{"type":"bool_type","loc":["body","force_refresh"],
  "msg":"Input should be a valid boolean","input":null}]}
```

**修复**：Java `PlanRequest` 加 `@JsonInclude(JsonInclude.Include.NON_NULL)`，不序列化 null 字段。Python 端字段有 `default=False`，收不到字段就用默认值。

## 12. 教务系统课表抓取

### 12.1 问题背景

学校教务系统技术栈：

| 组件 | 产品 | 域名 |
|---|---|---|
| 统一认证 | 苏文 CAS | `authserver.jxust.edu.cn` |
| 教务系统 | 金智教育 Wisedu | `jw.jxust.edu.cn/jsxsd/` |
| VPN 网关 | 深信服 aTrust | `jvpn.jxust.edu.cn` |

最初尝试纯 HTTP 自动登录（AES 加密密码 + ddddocr 验证码识别 + CAS 认证），CAS 认证本身调通了——AES 加密复刻正确、动态 salt 提取成功、ST ticket 获取正常。但最后一步请求教务课表页面时，深信服 aTrust VPN 网关拦截所有外网流量，返回 JavaScript 加载壳页面（5801 字节的 `<title>Loading...</title>`），Python httpx 无法执行 JS，永远拿不到真实课表 HTML。

**结论**：VPN 网关是纯 HTTP 爬虫的硬边界。决定只用 Playwright 真实浏览器方案。

### 12.2 方案：Playwright 浏览器自动化

**一个入口，一种方式**：`POST /courses/browser-import`

```
用户点"导入课表"
  → Playwright 启动 Chromium（有头模式，用户能看到）
  → 导航到 CAS 登录页
  → 用户手动输入学号/密码/验证码，完成登录
  → 系统每秒轮询检测：页面 DOM 中是否出现课表表格
  → 检测到后自动导航到课表详情页 xskb/xskb_list.do
  → 抓取 HTML → 解析 → 存入 courses.json
  → 关闭浏览器 → 返回结果
```

**为什么不让用户自己点课表链接？** 系统检测到课表表格后，会自动导航到 `xskb/xskb_list.do`（课表详情页），这个页面有完整的教师/地点/周次信息，比主页嵌入的简版课表信息更全。

### 12.3 核心代码：`tools/jwc.py`

文件只保留三个公开函数 + 配套解析器：

| 函数 | 用途 |
|---|---|
| `browser_login_and_parse(timeout_seconds)` | 浏览器自动化主流程，返回课程列表 |
| `_try_enrich_from_page(page)` | 四策略获取详情：导航详情页 → onclick → 链接 → 弹窗 |
| `parse_schedule_to_courses(html)` | HTML 表格 → 结构化课程数据 |

浏览器检测逻辑（每秒轮询）：

```python
while not timed_out:
    if "jsxsd" in page.url:  # 已登录教务
        has_table = await page.evaluate("""
            // 搜索 DOM 中是否有含 "节"+"一/二/三" 的 table
            // 同时检查 iframe
        """)
        if has_table:
            # 先尝试获取详情 → 兜底直接解析
            enriched = await _try_enrich_from_page(page)
            ...
```

**关键设计——不靠 URL 匹配**：课表嵌在教务主页里没有独立 URL，改为检查 DOM 中是否有包含课程数据的 `<table>` 元素。

### 12.4 HTML 解析器：金智教育表格的特殊格式

金智教育的课表 HTML 表格，每个课程格子里所有字段**拼在同一个字符串中**：

```
数据库系统概论1-16(周)1404数据库系统概论罗来俊1-16(周)[01-02节]1404(讲课)
```

格式规律：`课程名 + 周次 + 教室 + 课程名(重复) + 教师 + 周次 + [时间] + 教室 + (类型)`

部分格子用 `------` 分隔多个时间段（同一门课不同周次不同教室）。

**解析流程**（`parse_schedule_to_courses`）：

1. **构建网格**（`grid[r][c]`）：遍历所有 `<tr>`，记录每个 `<td>` 的 rowspan/colspan，填充到二维网格中，展开合并单元格。这解决了 rowspan 导致的列偏移问题

2. **表头 -> day 映射**：表头行中有"一/二/三..."的列 → 映射为 day=1..7

3. **逐格解析**：
   - 课程名：正则 `[\u4e00-\u9fffA-Za-z()（）]+?` 到第一个数字前停止
   - 教师：找紧挨着周次模式（`1-16(周)`）前的 2 个汉字，黑名单过滤课程名片段
   - 周次：`[\d,\-]+\(周\)|P\d+\(周\)` 匹配范围/枚举/P 标记三种格式
   - 地点：4 位教室号 / `三教`+"数字" / `田径场`

**教师提取的关键 trick**：用 `{2}` 而非 `{2,3}` 取紧挨周次前的 2 个汉字，避免贪婪匹配吞掉课程名最后一个字（如"系统"+"肖贺"→匹配到"统肖贺"而不是"肖贺"）。代价是 3 字教师名截为 2 字（罗来俊→来俊），可接受。

**不配对括号保护**：`re.sub(r'[()（）]$', '', course_name)` 会误删 `"大学英语(四)"` 的右括号。改为：仅当末尾是 `)` 且前面没有 `(` 时才删除。

## 13. 课表存储与周次筛选

### 13.2 学期周次计算

```python

def _get_current_week() -> int | None:
    start = parse_semester_start()
    today = date.today()
    return (today - start).days // 7 + 1

    """'1-16(周),P3(周)' → {1,2,...,16,3}"""
```

## 17. Day 3 总结：关键收获

### 17.1 技术收获

1. **VPN 网关是爬虫的硬边界**：深信服 aTrust 返回 JS 加载壳，纯 HTTP 无法突破。遇到 JS 渲染网关，不要死磕——直接上浏览器自动化

2. **Playwright 人类在环模式**：不是所有自动化都要全自动。用户手动登录 + 系统后台检测并抓取，绕过验证码/VPN/JS 渲染三大障碍。课表一学期导一两次，人工 1-2 分钟成本可接受

3. **HTML 解析的韧性设计**：金智教育的课表把所有字段拼成一个字符串，解析器不能假设标准格式。rowspan 网格展开 + 正则拆字段 + 黑名单过滤课程名片段，真实世界的 HTML 从来不会按预期结构化

4. **周次筛选的降级**：`_get_current_week()` 没配环境变量返回 `None`，`_is_week_active()` 收到 `None` 就放行所有课程。**可选功能不应成为 blocker**

5. **端口管理**：一天内 6 个 uvicorn 进程残留在不同端口，前端 Vite 代理指错导致 500/502。**每次测试前确认端口和版本对得上**

### 17.2 工作方法收获

6. **删代码比加代码更需要勇气**：CAS 自动登录的 AES 加密、验证码识别、ST ticket 提取代码都跑通了，但因为 VPN 阻断最终全删了。代码写得再好，跑不通就是负债

7. **前后端同步**：加了一个 API 端点，前端就需要对应的按钮、代理、超时、状态处理。改 Prompt 虽不影响接口签名，但用户体验完全不同
