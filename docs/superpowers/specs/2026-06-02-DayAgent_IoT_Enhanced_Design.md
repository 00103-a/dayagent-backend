# 2. 三阶段演进路线

# 5. IoT 数据链路优化

## 5.1 MQTT Topic 规范化

建议统一格式：

```text
dayagent/{device_id}/sensor/{sensor_type}
```

示例：

```text
dayagent/sim-01/sensor/environment
```

优势：

- 多设备扩展方便
- 后续真实硬件接入简单
- 支持设备分组

---

## 5.2 MQTT QoS 策略

当前：

- QoS = 0

优化建议：

| 场景 | QoS |
|---|---|
| 高频普通数据 | 0 |
| 关键告警 | 1 |
| 配置同步 | 2 |

---

## 5.3 设备状态机制

新增：

```json
{
  "device_id": "esp32-room-01",
  "status": "online",
  "battery": 82
}
```

支持：

- 在线检测
- 电量监控
- 离线告警

# 6. 数据存储优化

## 6.1 InfluxDB

存储：

- 时序环境数据
- 高频采样数据

保留策略：

| 数据 | 保存时间 |
|---|---|
| 原始数据 | 7 天 |
| 聚合数据 | 90 天 |
| AI 分析结果 | 长期 |

---

## 6.2 MySQL

用于：

- 用户配置
- 设备信息
- 提醒规则
- AI 个性设置

避免把时序数据写入 MySQL。

# 7. AI 分析层优化

## 7.1 双层分析架构

### 第一层：规则引擎

实时检测：

- CO₂ 超标
- 高温
- 光照异常

特点：

- 极低延迟
- 不依赖 LLM
- 高可靠

---

### 第二层：LLM 深度分析

每 10 分钟：

- 分析环境趋势
- 结合用户日程
- 生成个性化建议

示例：

> “你下午有连续课程，但房间空气质量正在变差，建议现在休息并通风。”

---

## 7.2 Prompt 上下文增强

注入：

- 今日课程
- 最近睡眠情况
- 环境变化趋势
- 用户历史习惯

形成长期记忆感。

# 11. 降级策略增强

| 故障 | 降级方案 |
|---|---|
| MQTT 挂掉 | 自动重连 + 缓存队列 |
| InfluxDB 挂掉 | 临时写 MySQL |
| LLM 失败 | 仅规则提醒 |
| Electron 崩溃 | 后端继续运行 |
| 模拟器停止 | 标记数据过期 |
| Docker 未启动 | 自动检测并警告 |
