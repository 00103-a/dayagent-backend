## 2. 坑 1：load_dotenv 路径错误

### 第一版修复（不完全）

```python
# main.py
from dotenv import load_dotenv
load_dotenv()  # 默认在当前工作目录找 .env
```

**问题**：用户从 `E:\dayagent` 启动 uvicorn，`load_dotenv()` 找的是 `E:\dayagent\.env`，但实际文件在 `E:\dayagent\agent_service\.env`。

**现象**：LLM 一直返回 401，错误信息里 Key 末尾是 `****lder`——匹配的是默认值 `sk-placeholder` 的末尾。说明 `.env` 完全没被读到。

### 最终修复

```python
# main.py
from pathlib import Path
from dotenv import load_dotenv

load_dotenv(Path(__file__).parent / ".env")
# __file__ = agent_service/main.py 的路径
# .parent   = agent_service/ 目录
# /.env     = agent_service/.env  ← 无论从哪个目录启动都正确
```

**教训**：`load_dotenv()` 不要裸调。用 `Path(__file__)` 相对定位，不依赖启动目录。

## 3. 坑 2：抓取串行改并行

### 问题

```python
# plan.py（修改前）
weather_info = await fetch_weather(req.location)     # 等 2s
chaoxing_tasks = await fetch_chaoxing_tasks()         # 再等 3s
course_info = await fetch_course_schedule()           # 再等 2s
# 总耗时: 2 + 3 + 2 = 7s
```

### 修复

```python
# plan.py（修改后）
import asyncio

weather_info, chaoxing_tasks, course_info = await asyncio.gather(
    fetch_weather(req.location),
    fetch_chaoxing_tasks(),
    fetch_course_schedule(),
)
# 总耗时: max(2, 3, 2) = 3s
```

### `asyncio.gather` 原理

```
串行:  天气 ─[2s]─→ 学习通 ─[3s]─→ 教务 ─[2s]─→ 完成 = 7s
并行:  天气 ─[2s]─→
       学习通 ───[3s]───→                           = 3s
       教务 ─[2s]─→
```

三个协程同时启动，总耗时约等于最慢的那个。`asyncio.gather` 只能并行 IO 等待（网络请求），不能并行 CPU 计算——Python GIL 限制，但 IO 等待时 GIL 会释放。

## 5. 坑 4：Java → Python JSON 字段名不匹配

### 问题

Java 的 `PlanRequest` 使用驼峰命名：

```java
public class PlanRequest {
    private String userId;           // 序列化成 "userId"
    private String yesterdaySummary; // 序列化成 "yesterdaySummary"
}
```

Python 的 Pydantic 模型使用下划线命名：

```python
class PlanRequest(BaseModel):
    user_id: str             # 期望 "user_id"
    yesterday_summary: str   # 期望 "yesterday_summary"
```

Java 发给 Python 的 JSON：
```json
{"userId": "1", "yesterdaySummary": "", ...}
```

FastAPI 收到后发现必填的 `user_id` 缺失，返回 422 校验失败。RestTemplate 把 422 响应当 PlanResponse 解析，结构对不上，抛异常 → Java 返回 500。

### 修复

`PlanRequest.java` 加 Jackson 注解：

```java
import com.fasterxml.jackson.annotation.JsonProperty;

@Data
@Builder
public class PlanRequest {
    @JsonProperty("user_id")
    private String userId;

    @JsonProperty("yesterday_summary")
    private String yesterdaySummary;

    private List<String> goals;    // 刚好同名，不用转
    private String location;       // 刚好同名，不用转
}
```

`goals` 和 `location` 两端命名一致，不需要 `@JsonProperty`。

## 6. 天气抓取：和风天气专属域名

### 初始代码的问题

初始代码写死了公共域名：

```python
"https://geoapi.qweather.com/v2/city/lookup"    # 城市搜索
"https://devapi.qweather.com/v7/weather/now"     # 实时天气
```

**现象**：城市搜索返回 404，天气接口返回 403 `Invalid Host`。

**根因**：和风天气改版后，每个开发者账号有专属 API Host（如 `mg33jqr7u9.re.qweatherapi.com`），公共域名不再接受免费 Key 的请求。

这个本应通过 WebSearch 提前发现——API 端点变更属于训练数据之外的实时信息。**后续所有外部 API 集成，先搜索最新文档再写代码。**

### 修复过程

**第一次尝试**：以为是 Key 没开通 API 产品，让用户去控制台重建项目。没用——Key 本身没问题，是域名不对。

**第二次尝试**：换成 wttr.in（免费天气 API，无需 Key）。能跑通但用户不满意——和风天气是国内标准方案，数据更准。

**第三次（最终修复）**：

1. `.env` 新增 `WEATHER_API_HOST` 变量：

```bash
WEATHER_API_HOST=mg33jqr7u9.re.qweatherapi.com   # 控制台→凭据→API Host
```

2. `weather.py` 从环境变量读 Host，函数内读取（非模块层，确保 `load_dotenv` 先执行）：

```python
async def fetch_weather(location: str) -> str:
    api_host = os.getenv("WEATHER_API_HOST", "https://api.qweather.com")
    if api_host and not api_host.startswith("http"):
        api_host = "https://" + api_host  # 控制台给的域名不带协议
```

3. 所有 API 调用使用 `api_host` 变量拼接：

```python
f"{api_host}/v2/city/lookup"     # 城市搜索
f"{api_host}/v7/weather/now"     # 实时天气
```

4. 城市搜索失败时降级为本地对照表（减少 API 调用失败的影响）：

```python
_CITY_ID_MAP = {
    "北京": "101010100",
    "上海": "101020100",
    # ... 20 个主要城市
}
```

### 另一个小 bug："东北风风"

`windDir` 返回的值本身就带"风"字（如"东北风"），模板又加了一个"风"→"东北风风"。修复：去掉模板里多余的"风"。

### 7.2 登录 ✅

```python
LOGIN_URL = "https://passport2.chaoxing.com/fanyalogin"

async def _login(client, username, password) -> bool:
    resp = await client.post(LOGIN_URL, data={
        "uname": username, "password": password, "t": "true"
    })
    return resp.json().get("status") is True
```

**验证结果**：`{"url":"http://i.mooc.chaoxing.com","status":true}` — 登录成功。

### 7.3 课程列表 ✅

```python
COURSE_LIST_URL = "https://mooc1-api.chaoxing.com/mycourse/backclazzdata"
```

**第一版解析代码（错误）**：

```python
courses.append({
    "course_id": item.get("course", {}).get("data", [{}])[0].get("id"),
    "class_id": item.get("key"),
    "name": item.get("course", {}).get("data", [{}])[0].get("name"),
})
```

**实际 API 返回结构**：`item` 自身就有 `id` 和 `name` 字段，不需要再嵌套一层 `course.data[0]`。

**修复**：

```python
courses.append({
    "course_id": item.get("id"),
    "class_id": ch.get("key"),
    "name": item.get("name", "未知课程"),
})
```

**验证结果**：成功抓到 8 门课——体育(四)、形势与政策(二)、信号与系统、幸福心理学、数字逻辑、Java程序设计、Python数据分析、数据库系统概论。

### 7.4 作业列表 ❌

```python
HOMEWORK_LIST_URL = "https://mooc1-api.chaoxing.com/work/homelist"
```

返回 404。尝试了多种替代路径（`/mooc-ans/work/homelist`、`/api/work/student/status` 等）全部 404。学习通改版后作业 API 路径已变更，需要重新抓包才能获取新地址。

**当前处理**：作业 API 挂了不阻塞整个流程。即使没有作业数据，课程列表本身也有价值。

### 7.5 结果拼装（改进后）

```python
def _format_result(homeworks, courses):
    lines = []
    # 课程列表始终输出，LLM 能知道用户这学期上什么课
    course_names = "、".join(c["name"] for c in courses)
    lines.append(f"本学期课程（共 {len(courses)} 门）：{course_names}")

    if homeworks:
        lines.append(f"学习通待完成：共 {len(homeworks)} 项")
        # ...
    else:
        lines.append("学习通作业列表暂不可用（API 变更），请手动确认各课程是否有作业截止。")
```

**改进前**：作业 API 挂了返回"暂无待完成作业"——LLM 误以为真的没作业。

**改进后**：返回课程名 + "作业列表暂不可用"——LLM 能据此提醒"检查信号与系统是否有作业截止"，比泛泛的"请手动检查"有用。

### 7.6 未覆盖的场景

- **设备认证**：新设备登录需要手机验证码
