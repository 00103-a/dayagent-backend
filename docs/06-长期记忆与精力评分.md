### Python 直连 MySQL 只读

```

    ↓ aiomysql (连接池 1-3)
```

连接信息从 `.env` 读取，与 Java `application.yml` 共享同一个数据库，但 Python 端只做只读查询。

## 三项分析详解

### 1. 精力趋势 (`_energy_trend`)

纯 SQL + 数学计算，不调用 LLM。

| 步骤 | 说明 |
|---|---|
| 查询 | 近 7 天 `daily_summary` 中 `mood_score IS NOT NULL` 的记录 |
| 均值 | 所有评分的算术平均 |
| 趋势 | 前半段 vs 后半段均值比较：差 > 0.5 为上升/下降，否则持平 |
| 连续低精力 | 遍历检测连续 `≤ 2` 的天数，≥ 2 天触发警告 |

输出示例：
```
【精力趋势】
近 5 天精力均值 3.2/5，趋势下降中 ↓
最近一次：2026-05-17 → 2/5
⚠ 连续 3 天精力偏低（≤2），今日建议减轻任务量
```

**边界情况**：无 mood_score 数据时返回"暂无精力数据"。

### 2. 规划 vs 总结对比 (`_plan_summary_contrast`)

查询近 7 天的 `daily_plan` 和 `daily_summary`，按日期配对输出原始文本。**不单独调 LLM**，而是将格式化后的对比数据注入主 Planner 的 LLM 调用中，由主 LLM 结合其他信息统一分析未完成事项。

输出示例：
```
【近期规划 vs 总结对比】
- 2026-05-17
  规划：📚 今日课程：高等数学 8:00-9:40...
  总结：完成了高数作业，但 LeetCode 没刷
- 2026-05-16
  规划：(无规划)
  总结：复习了线代第三章
```

**超长文本截断**：规划/总结超过 200 字自动截断，防止 Prompt 过长。

### 3. 目标进度检查 (`_goal_progress`)

纯 SQL + 日期计算，不调用 LLM。

| 步骤 | 说明 |
|---|---|
| 查询 | `goal` 表中 `status = 'active'` 的记录 |
| 进度 | `(今天 - start_date) / (end_date - start_date)` 转百分比 + 进度条 |
| 预警 | ≥ 80% 提示"即将到期"，≥ 100% 提示"已超期" |

输出示例：
```
【活跃目标进度】
- [weekly] 完成 Agent 层开发 [████████░░] 80% ⚠ 即将到期
- [monthly] 每天刷 2 道 LeetCode [███░░░░░░░] 30%
```

## 降级策略

| 场景 | 处理 |
|---|---|
| MySQL 连接失败 | `analyze_history()` 抛异常，`plan.py` 中此调用失败不影响主流程（asyncio.gather 之后单独 await，可加 try/except） |
| mood_score 全为 NULL | 返回"暂无精力数据" |
| 无规划/总结数据 | 返回"暂无规划/总结数据" |
| 无活跃目标 | 不输出目标板块 |
| aiomysql 未安装 | import 时报错，服务启动失败（依赖已在 requirements.txt） |

### 关于 analyze_history 的容错

当前 `plan.py` 中是直接 `await analyze_history(req.user_id)`，如果 DB 连不上会抛异常导致整个 `/generate-plan` 失败。如需降级，可改为：

```python
try:
    memory_hint = await analyze_history(req.user_id)
except Exception as e:
    print(f"[Memory] 分析失败: {e}")
    memory_hint = "长期记忆分析暂时不可用"
```
