## 1. 架构设计思路

### 1.1 为什么分成三个项目

不是刻意加复杂度。三种技术各有最擅长的领域：

```
浏览器 (Vue 3)          用户看到的界面
    ↓ HTTP
Java Spring Boot        业务大脑：用户管理、JWT 鉴权、数据持久化、定时任务
    ↓ HTTP（内部调用）
Python FastAPI          Agent 大脑：爬数据、调 LLM、生成规划
    ↓ HTTPS
外部服务（天气 API / LLM API）
```

**为什么 Java 不直接调 LLM，非要中间加一层 Python？**

学习通、教务系统的模拟登录涉及 HTML 解析、Cookie 管理、验证码识别，这些活在 Python 生态里用 `httpx` + `BeautifulSoup` 几十行就搞定了。Java 做同样的事（HttpClient + Jsoup）代码量翻倍且不稳定。反过来，用户管理、事务、定时任务、权限控制，Spring Boot 的生态比 Python 成熟得多。

结论：**Java 做业务网关，Python 做数据抓取和 AI 推理**，各司其职。

**为什么前端不直接调 Python？**

加一层 Java 做网关，JWT 鉴权只在一处做。如果前端直连 Python，Python 也得写一遍鉴权逻辑，重复劳动 + 安全风险。

### 1.2 数据流

```
用户点按钮
    → Vue 发 GET /api/plan?userId=1
    → Vite 代理到 Java :8080
    → PlanController 收到请求
    → AgentCallerService 查数据库（昨日总结 + 活跃目标）
    → AgentHttpClient 发 POST 到 Python :8000/generate-plan
    → plan.py 并行调天气/学习通/教务三个抓取工具
    → planner.py 拼 Prompt 调 LLM
    → 规划文本原路返回
    → Vue 展示
```

### 1.3 当前进度

第一阶段（今天）：骨架代码全部就位，整条链路跑通。学习通和教务是占位函数（直接返回"暂未接入"），天气 API 和 LLM 调用已实现但需要配 API Key。

### 2.2 `requirements.txt` — 依赖清单

```txt
fastapi>=0.115.0          # Web 框架，用装饰器定义接口路由
uvicorn[standard]>=0.34.0 # ASGI 服务器，运行 FastAPI 的引擎
httpx>=0.28.0             # 异步 HTTP 客户端（调天气 API 用）
pydantic>=2.10.0          # 数据校验框架，定义请求/响应结构体
python-dotenv>=1.0.0      # 从 .env 文件加载环境变量到 os.getenv()
openai>=1.59.0            # OpenAI SDK，兼容一切 OpenAI 格式的 LLM（豆包/通义/Claude）
```

**为什么版本号用 `>=` 而不是 `==`？**

第一天踩的坑：你用的是 Python 3.14，`pydantic==2.10.4` 依赖的 `pydantic-core==2.27.2` 没有 Python 3.14 的预编译 `.whl` 包，pip 只能从 Rust 源码编译，而 Windows 上缺 Rust 工具链导致安装失败。改用 `>=` 后 pip 自动选了有预编译包的最新版（2.13.4），问题解决。

**教训**：新项目不要锁死版本号，除非明确知道需要某个特定版本。`>=` 给 pip 足够的兼容空间。

### 2.4 `schemas/plan_request.py` — Pydantic 数据模型

```python
from pydantic import BaseModel, Field

class PlanRequest(BaseModel):
    """Java 端调用 /generate-plan 时传入的数据"""
    user_id: str = Field(..., description="用户 ID")
    yesterday_summary: str = Field(default="", description="昨日总结")
    goals: list[str] = Field(default_factory=list, description="当前活跃目标列表")
    location: str = Field(default="北京", description="城市，用于查天气")

class PlanResponse(BaseModel):
    """返回给 Java 端的规划结果"""
    plan: str = Field(..., description="今日规划正文")
    priorities: list[str] = Field(default_factory=list, description="优先级列表")
    warnings: list[str] = Field(default_factory=list, description="预警提醒，如作业截止")
```

**逐行解释：**

- `BaseModel`：Pydantic 的核心类。继承它之后，FastAPI 会自动做三件事：
  - **请求校验**：如果 Java 发来的 JSON 少字段或类型不对，直接返回 422 错误，不污染业务代码
  - **自动文档**：Swagger UI (`/docs`) 里自动生成请求体示例
  - **序列化/反序列化**：Python dict ↔ JSON 字符串自动转换

- `Field(...)`：`...` 是 Python 的 Ellipsis 字面量，在这里表示**必填字段**，没有默认值

- `Field(default="")`：有默认值的字段，调用方可以不传

- `Field(default_factory=list)`：**为什么不用 `default=[]`？** 这是 Python 的经典坑。如果写 `default=[]`，所有请求会共享同一个 list 对象，请求 A 修改了列表，请求 B 也会看到修改后的值。`default_factory=list` 每次调用都创建新的空列表，线程安全。

**这两个类同时扮演了两个角色**：

1. **文档角色**：在 Swagger 页面上生成输入/输出说明
2. **校验角色**：运行时自动检查数据格式

### 2.5 `routers/plan.py` — 核心接口

```python
from fastapi import APIRouter
from agent_service.schemas.plan_request import PlanRequest, PlanResponse
from agent_service.tools.weather import fetch_weather
from agent_service.tools.chaoxing import fetch_chaoxing_tasks
from agent_service.tools.jwc import fetch_course_schedule
from agent_service.agents.planner import generate_plan
from agent_service.agents.memory import analyze_history

router = APIRouter()

@router.post("/generate-plan", response_model=PlanResponse)
async def generate_plan_endpoint(req: PlanRequest) -> PlanResponse:
    # 1. 拉取多源数据
    weather_info = await fetch_weather(req.location)
    chaoxing_tasks = await fetch_chaoxing_tasks()
    course_info = await fetch_course_schedule()

    # 2. 长期记忆分析（目前是空函数）
    memory_hint = await analyze_history(req.user_id)

    # 3. 调 LLM
    result = await generate_plan(
        weather_info=weather_info,
        chaoxing_tasks=chaoxing_tasks,
        course_info=course_info,
        yesterday_summary=req.yesterday_summary,
        goals=req.goals,
        memory_hint=memory_hint,
    )
    return result
```

**逐行解释：**

- `APIRouter()`：FastAPI 的路由器。相当于 Flask 的 Blueprint 或 Spring 的 `@RestController`。使用 Router 而不是直接在 `main.py` 写所有接口，是为了**接口和入口分离**——`main.py` 只管启动，`plan.py` 只管业务逻辑

- `@router.post("/generate-plan", response_model=PlanResponse)`：装饰器三要素：
  - `"/generate-plan"`：URL 路径
  - `post`：HTTP 方法
  - `response_model=PlanResponse`：告诉 FastAPI 响应格式，用于自动文档和序列化

- `async def`：Python 异步函数。`async`/`await` 的核心价值在于 **IO 密集操作不阻塞线程**——调天气 API 时，等待网络返回的几百毫秒里，线程可以处理其他请求

- **代码里的问题（故意的）**：三个数据抓取是串行的：
  ```python
  weather_info = await fetch_weather(...)      # 等这个返回
  chaoxing_tasks = await fetch_chaoxing_tasks() # 才执行这个
  course_info = await fetch_course_schedule()   # 最后执行这个
  ```
  它们互不依赖（不需要天气结果才能查学习通），应该用 `asyncio.gather` 并行：
  ```python
  weather, chaoxing, course = await asyncio.gather(
      fetch_weather(req.location),
      fetch_chaoxing_tasks(),
      fetch_course_schedule(),
  )
  ```
  三者的总耗时 = max(各自耗时)，而不是 sum(各自耗时)。这是后续优化的点。

### 2.6 `tools/weather.py` — 和风天气 API

```python
import os
import httpx

async def fetch_weather(location: str) -> str:
    api_key = os.getenv("WEATHER_API_KEY", "")
    if not api_key:
        return "天气数据暂不可用"

    async with httpx.AsyncClient() as client:
        # Step 1: 城市搜索（中文名 → city_id）
        geo_resp = await client.get(
            "https://geoapi.qweather.com/v2/city/lookup",
            params={"location": location, "key": api_key},
            timeout=5.0,
        )
        if geo_resp.status_code != 200 or not geo_resp.json().get("location"):
            return f"{location}：天气数据获取失败"
        city_id = geo_resp.json()["location"][0]["id"]

        # Step 2: 用 city_id 查实时天气
        weather_resp = await client.get(
            "https://devapi.qweather.com/v7/weather/now",
            params={"location": city_id, "key": api_key},
            timeout=5.0,
        )
        if weather_resp.status_code != 200:
            return f"{location}：天气数据获取失败"

        now = weather_resp.json()["now"]
        return (
            f"{location}今日天气：{now['text']}，"
            f"温度 {now['temp']}°C，"
            f"体感温度 {now['feelsLike']}°C，"
            f"{now['windDir']}风 {now['windScale']} 级，"
            f"相对湿度 {now['humidity']}%"
        )
```

**逐行解释：**

- `os.getenv("WEATHER_API_KEY", "")`：从环境变量读 API Key，第二个参数是默认值（找不到就返回空字符串）

- **为什么每次调用都创建新的 `AsyncClient()`？** `async with` 保证用完自动关闭连接。其实更好的做法是创建一个全局的 `AsyncClient()` 实例复用连接池，但早期项目先这样写——简单优先，优化在后面

- `timeout=5.0`：5 秒超时。不加超时的话，如果和风天气服务器挂了，请求会一直挂着，最终拖死整个接口。**所有外部 HTTP 调用都必须加超时**

- **为什么返回自然语言字符串而不是 JSON？** 目的不是给人看结构化数据，而是要喂给 LLM 的 Prompt。直接返回自然语言描述，Prompt 里可以直接拼进去，不用再做一次格式转换

- **两段式 API 调用**：和风天气的设计是先查 city ID（geoapi），再用 city ID 查天气（devapi）。因为天气 API 不接受城市名，只接受 location ID

### 2.9 `agents/planner.py` — LLM 调用核心

```python
import os
from pathlib import Path
from openai import AsyncOpenAI
from agent_service.schemas.plan_request import PlanResponse

_PROMPT_DIR = Path(__file__).parent.parent / "prompts"

def _load_prompt(filename: str) -> str:
    path = _PROMPT_DIR / filename
    if path.exists():
        return path.read_text(encoding="utf-8")
    return ""

async def generate_plan(
    weather_info: str,
    chaoxing_tasks: str,
    course_info: str,
    yesterday_summary: str,
    goals: list[str],
    memory_hint: str,
) -> PlanResponse:
    system_prompt = _load_prompt("planner_prompt.txt")
    goals_text = "\n".join(f"- {g}" for g in goals) if goals else "暂无"

    user_message = f"""## 今日数据

### 天气
{weather_info}

### 学习通作业
{chaoxing_tasks}

### 教务课表
{course_info}

### 昨日总结
{yesterday_summary or "暂无"}

### 当前目标
{goals_text}

### 长期记忆分析
{memory_hint or "暂无"}

请根据以上数据，生成今日的行动规划。"""

    client = AsyncOpenAI(
        base_url=os.getenv("LLM_BASE_URL", "https://api.openai.com/v1"),
        api_key=os.getenv("LLM_API_KEY", "sk-placeholder"),
    )
    model = os.getenv("LLM_MODEL", "gpt-3.5-turbo")

    try:
        resp = await client.chat.completions.create(
            model=model,
            messages=[
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": user_message},
            ],
            temperature=0.7,
            max_tokens=2000,
            timeout=30.0,
        )
        content = resp.choices[0].message.content or "规划生成失败"
    except Exception as e:
        content = f"LLM 调用失败：{e}"

    return PlanResponse(plan=content, priorities=[], warnings=[])
```

**逐行解释：**

- `Path(__file__).parent.parent / "prompts"`：`__file__` 是当前文件路径 → `.parent` 是 `agents/` 目录 → `.parent` 是 `agent_service/` 根目录 → `/"prompts"` 拼出 `prompts/` 目录的路径。用 `pathlib` 而不是字符串拼接，跨平台不会出错（Windows 用 `\`，Linux 用 `/`）

- `_load_prompt` 以下划线开头：Python 约定，`_` 前缀表示"模块内部使用，外部不要调"

- **Prompt 为什么要从文件加载，不硬编码在代码里？**
  1. Prompt 会频繁调整（换措辞、加规则、改输出格式），改 txt 文件比改 Python 代码安全
  2. 非程序员（产品经理）也能看和改 Prompt
  3. 版本控制更清晰：`git log prompts/planner_prompt.txt` 只看 Prompt 变更历史

- `AsyncOpenAI`：用的是 OpenAI 官方的异步客户端。**豆包/通义/Claude 都兼容 OpenAI 的 `/v1/chat/completions` 格式**，所以用 OpenAI 的 SDK 设置 `base_url` 指向不同的地址就能通吃所有模型

- `temperature=0.7`：控制创意程度。0 = 完全确定性（每次相同输入输出相同），1 = 最随机。规划任务 0.7 比较合适——需要一些创意，但不能太跳脱

- `max_tokens=2000`：限制输出长度，防止 LLM 写一篇论文出来

- `try/except`：LLM API 可能挂（欠费、超时、限流），必须 catch。这里把异常信息直接写进返回内容，用户至少看到"出了什么事"，而不是白屏

### 2.11 `prompts/planner_prompt.txt` — System Prompt

```text
你是一个专业的个人日程规划助手。你会根据用户当日的天气、学习任务、
课程安排、昨日总结和长期目标，生成一份清晰、可执行的当日行动计划。

## 规划原则
1. **优先级排序**：有截止日期的任务排在最前面
2. **结合天气**：雨天建议室内学习，晴天可以建议户外运动
3. **结合课表**：避开上课时间段安排学习任务
4. **参考昨日**：如果用户昨天某任务未完成，今天优先补上
5. **符合目标**：每天至少要有一件事能推进周/月目标

## 输出格式
直接输出今日规划的纯文本，分以下小节：
- ☀️ 今日概览（1-2句）
- 📋 优先级任务（编号列表，每条说明原因）
- ⏰ 建议时间安排（按时间块）
- ⚠️ 提醒事项

不要用 JSON 格式，用户直接看这份文本。
```

**设计要点：**

- **给 LLM 角色定位**（第一句）："你是一个专业的xxx"，这种方式比直接下命令效果好得多
- **明确约束**（规划原则 5 条）：不是模糊的"帮用户规划"，而是具体的排序规则
- **固定输出格式**：有了统一模板，以后想做前端解析展示也方便
- **分文件存放**：改 Prompt 不影响代码逻辑，IDE 里改 txt 不会触发 Python 语法检查报错

### 2.12 `main.py` — FastAPI 入口

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI
from agent_service.routers import plan

@asynccontextmanager
async def lifespan(app: FastAPI):
    print("Agent Service 启动成功")
    yield
    # 关闭时做的事（比如关数据库连接池）

app = FastAPI(
    title="DayAgent Agent Service",
    description="多源数据采集 + LLM 规划生成",
    version="0.1.0",
    lifespan=lifespan,
)

app.include_router(plan.router)

@app.get("/health")
async def health_check():
    return {"status": "ok"}
```

**逐行解释：**

- `lifespan`：FastAPI 的生命周期管理。`yield` 之前的代码在启动时执行（初始化连接池等），`yield` 之后在关闭时执行（释放资源）。目前只是打印了一行日志

- `app.include_router(plan.router)`：把 `routers/plan.py` 里定义的所有路由注册到应用上。**这是 FastAPI 的推荐做法**——入口文件保持简洁，具体路由分散到各自的 Router 文件里

- `/health` 端点：Java 端的健康检查会定期调这个接口。如果返回非 200，Java 端就知道 Python 挂了，启用降级策略。**这是微服务的基本素养**——每个服务都提供健康检查

- **为什么没有 `load_dotenv()`？** 漏了。应该在文件最前面加一行：
  ```python
  from dotenv import load_dotenv
  load_dotenv()
  ```
  否则 `.env` 文件里的变量不会加载到 `os.getenv()` 里。需要后续补上。

### 3.2 Spring Boot 分层架构说明

Java 端采用了经典的**三层架构**：

```
Controller 层  → 接收 HTTP 请求，参数校验，调 Service，返回响应
    ↓
Service 层     → 业务逻辑：查数据库、组装数据、调外部服务
    ↓
Mapper 层      → 数据库操作（MyBatis Plus 自动生成 CRUD）
    ↓
Entity 层      → 数据库表对应的 Java 类
```

**每层只做自己的事，不越界**。比如 Controller 绝不直接写 SQL，Mapper 绝不处理业务逻辑。

**`spring-boot-starter-parent` 是什么？**

### 3.5 `DayAgentApplication.java` — 启动类


三行注解，三个意义：

### 3.6 Entity 层 — 数据库表映射

以 `User.java` 为例：

```java
@Data
@TableName("user")                  // 对应数据库的 user 表
public class User {
    @TableId(type = IdType.AUTO)    // 主键，自增
    private Long id;
    private String username;
    private String password;        // BCrypt 加密存储，绝不存明文
    private String wechatWorkId;    // 用于企业微信推送
    private LocalDateTime createdAt;
}
```

**注解解释：**

- `@Data`：Lombok 的瑞士军刀。编译时自动生成：`getXxx()`、`setXxx()`、`toString()`、`equals()`、`hashCode()`。没有 Lombok，每个 Entity 要多写 50+ 行机械代码
- `@TableName("user")`：MyBatis Plus 默认用类名小写当表名，`User` → `user`，刚好匹配。但如果类名是 `DailySummary`，默认找 `daily_summary` 表（驼峰转下划线），所以写不写都行，**写了更明确**
- `@TableId(type = IdType.AUTO)`：告诉 MyBatis Plus 这个字段是数据库自增主键。Insert 时不传 id，数据库自动生成

其余三个 Entity（`Goal`、`DailySummary`、`DailyPlan`）结构完全一致，只是字段不同，不重复解释。

```java
@Mapper
public interface UserMapper extends BaseMapper<User> {
```

```java
userMapper.insert(user);                    // 插入
```

### 3.8 Service 层 — 业务逻辑

#### 3.8.1 `AgentHttpClient.java` — HTTP 调用封装

```java
@Service
@RequiredArgsConstructor
public class AgentHttpClient {

    private final RestTemplate restTemplate;

    @Value("${agent.base-url}")
    private String agentBaseUrl;       // 从 yml 注入 = http://localhost:8000

    public PlanResponse callGeneratePlan(PlanRequest request) {
        String url = agentBaseUrl + "/generate-plan";
        try {
            return restTemplate.postForObject(url, request, PlanResponse.class);
        } catch (Exception e) {
            // 降级：Python 挂了不阻塞主流程
            PlanResponse fallback = new PlanResponse();
            fallback.setPlan("今日规划暂时无法生成，请稍后重试。");
            fallback.setPriorities(List.of());
            fallback.setWarnings(List.of("Agent 服务不可用：" + e.getMessage()));
            return fallback;
        }
    }
}
```

**关键设计决策——降级策略：**

Python 服务可能挂（依赖外部 API、LLM 限流、内存溢出），如果 Java 端也跟着抛异常，前端直接白屏。正确的做法是 **catch 异常 + 返回降级数据**：告诉用户"生成失败了，先做昨天没做完的事"，至少不是白屏。

**为什么 `10s` 超时？** 最初设计文档写了超时 10s，但这个版本还没配置。后续在 `RestTemplateConfig` 里加：
```java
@Bean
public RestTemplate restTemplate() {
    RestTemplate template = new RestTemplate();
    HttpComponentsClientHttpRequestFactory factory = new HttpComponentsClientHttpRequestFactory();
    factory.setConnectTimeout(5000);    // 连上等 5 秒
    factory.setReadTimeout(10000);      // 返回等 10 秒
    template.setRequestFactory(factory);
    return template;
}
```

- `@RequiredArgsConstructor`：Lombok 注解，为所有 `final` 字段生成构造函数。Spring 自动通过构造函数注入 Bean。**为什么用构造器注入而不是 `@Autowired`？** 构造器注入是 Spring 官方推荐的方式——依赖在构造时就被确定，不会出现 NPE，也方便写单元测试时手动 new 对象传 mock

- `@Value("${agent.base-url}")`：从 `application.yml` 读取 `agent.base-url` 的值注入

#### 3.8.2 `AgentCallerService.java` — 业务编排

```java
@Service
@RequiredArgsConstructor
public class AgentCallerService {

    private final AgentHttpClient agentHttpClient;
    private final GoalMapper goalMapper;
    private final DailySummaryMapper dailySummaryMapper;

    public PlanResponse generatePlan(Long userId, String location) {
        // 1. 查用户昨天的总结
        DailySummary yesterday = dailySummaryMapper.selectOne(
            new LambdaQueryWrapper<DailySummary>()
                .eq(DailySummary::getUserId, userId)
                .orderByDesc(DailySummary::getSummaryDate)
                .last("LIMIT 1")
        );

        // 2. 查活跃目标
        List<Goal> activeGoals = goalMapper.selectList(
            new LambdaQueryWrapper<Goal>()
                .eq(Goal::getUserId, userId)
                .eq(Goal::getStatus, "active")
        );

        // 3. 组装请求
        PlanRequest request = PlanRequest.builder()
                .userId(String.valueOf(userId))
                .yesterdaySummary(yesterday != null ? yesterday.getContent() : "")
                .goals(activeGoals.stream().map(Goal::getContent).toList())
                .location(location)
                .build();

        return agentHttpClient.callGeneratePlan(request);
    }
}
```

**`LambdaQueryWrapper` 是什么？**

MyBatis Plus 提供的类型安全查询构造器。它的价值在于**编译时检查**：

```java
// 类型安全：如果 Goal 没有 getStatus 方法，IDE 直接标红
.eq(Goal::getStatus, "active")

// 对比传统 MyBatis 写法（字符串拼字段名，写错了编译不报错，运行时才炸）：
.eq("status", "active")  // 字段名拼成 "satuts" 也编译通过
```

**`PlanRequest.builder()` 是什么？**

Lombok 的 `@Builder` 注解生成的建造者模式代码。普通写法：
```java
PlanRequest req = new PlanRequest();
req.setUserId("1");
req.setLocation("北京");
```

Builder 写法（更清晰，支持链式调用）：
```java
PlanRequest req = PlanRequest.builder()
    .userId("1")
    .location("北京")
    .build();
```

#### 3.8.3 `PlanRequest.java` 和 `PlanResponse.java`

这两个是 DTO（Data Transfer Object，数据传输对象），和 Python 端的 `schemas/plan_request.py` 字段一一对应。**两边各写一遍**是因为 Java 和 Python 是独立的服务，语言不通，只能靠约定保持字段名一致。

#### 3.8.4 `PushService.java` — 企微推送（占位）

```java
@Service
public class PushService {
    public void pushMessage(Long userId, String content) {
        // TODO: 第二阶段实现
        System.out.println("推送消息给用户 " + userId + "：" + content);

```

现在是 `println` 占位，第二阶段换成企微 Webhook 的 HTTP 调用。

#### 3.8.5 `ScheduleService.java` — 定时任务

```java
@Slf4j
@Component
public class ScheduleService {

    @Scheduled(cron = "0 0 8 * * ?")
    public void morningPlanGeneration() {
        log.info("开始执行每日早8点规划任务...");
        // TODO: 遍历所有活跃用户，逐个生成规划并推送
        log.info("每日规划任务执行完毕");
    }
}
```

**Cron 表达式 `0 0 8 * * ?` 逐位解释**：

| 位 | 值 | 含义 |
|---|---|---|
| 秒 | 0 | 第 0 秒 |
| 分 | 0 | 第 0 分 |
| 时 | 8 | 早上 8 点 |
| 日 | * | 每一天 |
| 月 | * | 每一月 |
| 周 | ? | 不指定（和日字段冲突时用 ? 避开） |

**Java 的 Cron 有 6 位（秒 分 时 日 月 周），Linux 的 Cron 只有 5 位**，这是常见的坑。

`@EnableScheduling` 在启动类上已开启，这个定时任务会在每天早上 8:00:00 自动触发。

### 3.9 Controller 层 — REST 接口

以 `PlanController.java` 为例：

```java
@RestController
@RequestMapping("/api")
@RequiredArgsConstructor
public class PlanController {

    private final AgentCallerService agentCallerService;

    @GetMapping("/plan")
    public PlanResponse getPlan(
            @RequestParam Long userId,
            @RequestParam(defaultValue = "北京") String location
    ) {
        return agentCallerService.generatePlan(userId, location);
    }
}
```

**注解解释：**

- `@RestController`：`@Controller` + `@ResponseBody`。每个方法的返回值直接序列化为 JSON，不经过视图解析器
- `@RequestMapping("/api")`：这个 Controller 下所有接口都以 `/api` 开头
- `@GetMapping("/plan")`：完整路径 = `/api` + `/plan` = `GET /api/plan`
- `@RequestParam(defaultValue = "北京")`：前端可以不传 location 参数，默认北京

Controller 层非常薄——它不做任何业务逻辑，只是**参数接收 + 调 Service + 返回结果**。这是正确的分层：Controller 是"胶水层"而非"逻辑层"。

其余三个 Controller（`UserController`、`GoalController`、`SummaryController`）都是同样的模式：接收参数 → 调 Mapper → 返回结果。当前 `login` 方法是占位，第二/三阶段补充 JWT 校验。

### 3.10 Config 层 — 配置类

#### `RestTemplateConfig.java`

```java
@Configuration
public class RestTemplateConfig {
    @Bean
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
```

`@Bean` 方法返回的对象由 Spring 容器管理（单例，整个应用共享一个 RestTemplate 实例）。其他地方通过构造器注入就能拿到。

#### `SecurityConfig.java`

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public BCryptPasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .authorizeHttpRequests(auth -> auth
                .anyRequest().permitAll()  // 第一阶段全部放行
            );
        return http.build();
    }
}
```

**为什么关掉 CSRF？** CSRF 防御需要前端每次请求都带一个 Token，REST API + JWT 的场景下不需要（JWT 本身已防 CSRF）。不关的话所有 POST/PUT/DELETE 都会被拦。

**`anyRequest().permitAll()`**：第一阶段不校验权限，所有接口公开访问。后续改为：
```java
.requestMatchers("/api/user/login", "/api/user/register").permitAll()
.anyRequest().authenticated()
```

### 4.5 `api/plan.js` — HTTP 调用封装

```javascript
import axios from 'axios'

export async function fetchPlan(userId = 1, location = '北京') {
  const { data } = await axios.get('/api/plan', {
    params: { userId, location },    // → /api/plan?userId=1&location=北京
  })
  return data
  // data 结构：{ plan: "...", priorities: [...], warnings: [...] }
}
```

**为什么把 axios 调用抽成一个函数而不是在组件里直接写？**

1. **复用**：多个组件需要调同一个接口时不用重复
2. **隔离**：组件不关心请求细节（base URL、参数拼法、错误处理），只管调函数
3. **可测试**：可以单独 mock 这个函数，不用启动真正的后端

URL 写的是 `/api/plan` 而不是 `http://localhost:8080/api/plan`，因为 Vite 的代理会把 `/api` 前缀转发到 Java。

### 4.6 `App.vue` — 根组件（SFC 三合一）

Vue 的 `.vue` 文件 = `<script>` + `<template>` + `<style>` 三块放在一个文件里，叫**单文件组件（SFC）**。

#### 脚本部分

```javascript
<script setup>
import { ref } from 'vue'
import { fetchPlan } from './api/plan.js'

const plan = ref('')          // 响应式字符串
const priorities = ref([])     // 响应式数组
const warnings = ref([])
const loading = ref(false)

async function loadPlan() {
  loading.value = true
  try {
    const data = await fetchPlan()
    plan.value = data.plan
    priorities.value = data.priorities
    warnings.value = data.warnings
  } catch (e) {
    plan.value = '获取规划失败：' + e.message
  } finally {
    loading.value = false
  }
}
</script>
```

**`<script setup>` vs 普通 `<script>`：**

Vue 3.2 引入的语法糖。`<script setup>` 里定义的变量和函数自动暴露给模板，不需要 `return`。相当于：
```javascript
// 没有 setup 语法糖时的写法
export default {
  setup() {
    const plan = ref('')
    // ...
    return { plan, priorities, warnings, loading, loadPlan }  // 手动 return
  }
}
// 有 setup 语法糖 → 省掉 export default 和 return
```

**`ref` 是什么？**

Vue 3 的响应式 API。`ref('')` 创建了一个响应式字符串，当它的值改变时，模板里用到它的地方自动重新渲染。

**注意 `.value`**：在 JavaScript 里读/写 ref 的值要加 `.value`（`plan.value = data.plan`），但在模板里自动解包，直接写 `{{ plan }}` 就行。这是新手最容易搞混的地方。

#### 模板部分

```html
<template>
  <button @click="loadPlan" :disabled="loading">
    {{ loading ? '生成中...' : '获取今日规划' }}
  </button>

  <div v-if="warnings.length" class="warnings">
    <p v-for="w in warnings" :key="w">⚠️ {{ w }}</p>
  </div>

  <div v-if="plan" class="plan-content">
    <pre>{{ plan }}</pre>
  </div>
</template>
```

**Vue 指令速查：**

| 语法 | 含义 |
|---|---|
| `@click="loadPlan"` | 点击时执行 `loadPlan` 函数 |
| `:disabled="loading"` | 动态绑定 disabled 属性（loading 为 true 时禁用按钮） |
| `{{ loading ? '...' : '...' }}` | 插值表达式，三元运算 |
| `v-if="condition"` | 条件渲染（false 时不生成 DOM） |
| `v-for="w in warnings"` | 列表渲染 |
| `:key="w"` | 循环中每个元素的唯一标识，Vue 用它做 diff 优化 |

#### 样式部分

```css
<style>
.container { max-width: 640px; margin: 60px auto; ... }
button { background: #4f46e5; ... }
button:disabled { background: #a5b4fc; cursor: not-allowed; }
</style>
```

CSS 写在 `.vue` 文件里默认是**全局样式**（不加 `scoped`）。目前只有一个页面所以无所谓，后面组件多了需要加 `<style scoped>` 防止样式污染。

## 5. 工程配置文件

### 5.1 `.gitignore`

```gitignore
.env              # 环境变量（含 API Key、密码）
__pycache__/      # Python 编译缓存
*.pyc
.venv/
venv/
target/           # Maven 编译输出
*.class
node_modules/     # npm 依赖（一个黑洞，不应进 git）
dist/             # 前端打包输出
.idea/            # IDEA 配置（每个人 IDE 设置不同）
```

**核心原则**：一切能**从源码重新生成**的东西都不进 Git（编译产物、依赖包、IDE 个人设置）。只有环境变量模板（`.env.example`）进 Git，真 `.env` 不进来。

### 5.2 `schema.sql` — 建表脚本

四张表的设计逻辑：

**user 表**：最基础的用户表。`wechat_work_id` 是企微推送的目标 ID（第二阶段用）。密码存的是 BCrypt 加密后的哈希，**永远不存明文**。

**daily_summary 表**：每天晚上用户写的总结。`idx_user_date` 索引是为了快速查"某用户昨天的总结"（`AgentCallerService` 里用的就是这个索引）。

**goal 表**：周目标 / 月目标。`type` 字段用 ENUM 而不是字符串，因为只有两种类型，ENUM 可以防止写错值（比如写成 `'weekley'` 会直接报错）。

**daily_plan 表**：当天规划的缓存。`raw_data` 是 JSON 类型，存天气数据、学习通数据等的原始快照——将来复盘时可以看"当时是基于什么信息做出的这个规划"。

**为什么不每天直接删了重建？** 保留历史规划可以分析：规划完成率、用户行为变化趋势。这些数据是第三阶段做长期记忆的原材料。

## 6. 数据流向全景图

### 6.1 一次"获取规划"的完整调用链

```
1. 用户点击按钮
   ↓
2. App.vue: loadPlan()
   调用 fetchPlan() → axios.get('/api/plan?userId=1&location=北京')
   ↓
3. Vite Dev Server（:3000）
   检测到 /api 前缀 → 代理转发到 http://localhost:8080
   ↓
4. Java: PlanController.getPlan()
   接收参数 userId=1, location="北京"
   调用 agentCallerService.generatePlan(1L, "北京")
   ↓
5. Java: AgentCallerService.generatePlan()
   - dailySummaryMapper.selectOne()  → 查昨天总结
   - goalMapper.selectList()         → 查活跃目标
   - PlanRequest.builder()...build() → 组装请求体
   - agentHttpClient.callGeneratePlan(request)
   ↓
6. Java: AgentHttpClient.callGeneratePlan()
   restTemplate.postForObject("http://localhost:8000/generate-plan", request, ...)
   ↓  (HTTP POST, JSON body)
7. Python: plan.generate_plan_endpoint()
   - await fetch_weather("北京")        → "北京今日天气：晴，温度25°C..."
   - await fetch_chaoxing_tasks()       → "学习通数据暂未接入"
   - await fetch_course_schedule()      → "教务课表暂未接入"
   - await analyze_history("1")         → ""
   - await generate_plan(...)           → 拼 Prompt, 调 LLM
   ↓
8. Python: planner.generate_plan()
   - 加载 prompts/planner_prompt.txt 作为 system prompt
   - 拼多源数据作为 user message
   - AsyncOpenAI.chat.completions.create()
   ↓  (HTTPS, OpenAI 兼容格式)
9. LLM API（豆包/通义/Claude）
   返回规划文本
   ↓
10. 原路返回
    Python → Java → Vite → 浏览器
    ↓
11. App.vue: 数据赋给 ref → Vue 自动更新 DOM → 用户看到规划
```

### 6.2 各环节的失败处理

| 环节 | 失败原因 | 当前处理 |
|---|---|---|
| 天气 API | 没配 Key / 超时 | 返回"天气数据暂不可用" |
| 学习通 | 未实现 | 返回"暂未接入" |
| LLM API | 欠费/超时/限流 | catch 异常，返回错误提示 |
| Python 整体挂掉 | 进程崩溃 | Java catch 异常，返回降级数据 |
| Java 挂掉 | 进程崩溃 | 前端 catch，显示错误信息 |

**整个链路没有单点故障会让用户看到白屏**，每一层都有降级策略。
