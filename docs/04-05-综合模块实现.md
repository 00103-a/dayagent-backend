## 第一部分：个性化新闻模块

### 架构设计

```
/generate-plan 请求
    ↓
asyncio.gather (并行)
    ├── fetch_weather()
    ├── fetch_chaoxing_tasks()
    ├── get_today_courses()
    └── get_personalized_news(goals, yesterday_summary)  ← 新增
            ↓
        1. extract_keywords_llm(goals, summary)
           └── 降级: extract_keywords_basic()  无 LLM 时走规则提取
        2. fetch_raw_news()
           └── 数据源未配置时返回空列表（不阻塞主流程）
        3. filter_news_by_llm(raw_news, keywords)
           └── 降级: 返回前 3 条原始新闻
        4. 格式化文本
    ↓
generate_plan(..., news_info=...) → LLM 生成最终汇报
```

### 关键设计决策

#### 1. 降级策略（三层容错）

| 环节 | 首选方案 | 降级方案 | 兜底 |
|---|---|---|---|
| 关键词提取 | `extract_keywords_llm()` — DeepSeek 提取 3~8 个关键词 | `extract_keywords_basic()` — 正则分词 + 去停用词 | `["科技", "校园", "考试"]` |
| 新闻拉取 | 调用配置的数据源 API | 返回空列表 | 不影响主流程 |
| 新闻筛选 | `filter_news_by_llm()` — LLM 按索引筛选 | 返回前 3 条原始新闻 | 空列表 |

#### 2. 数据源预留

`fetch_raw_news()` 当前返回空列表，`_parse_news_response()` 预做了 NewsAPI / 聚合数据 / 天行数据等多种常见返回格式的解析逻辑。接入时只需：
1. 在 `.env` 配置 `NEWS_API_KEY` 和 `NEWS_API_URL`
2. 根据实际 API 微调 `_parse_news_response()` 中的字段映射

#### 3. LLM 筛选策略

将 30 条新闻按索引编号发送给 LLM，LLM 返回相关新闻的索引，程序按索引取回。避免 LLM 输出新闻内容可能产生的幻觉。

## 第二部分：快递查询模块

### 架构

```
前端 (Vue 3)
    ↓ POST/GET/DELETE /api/parcel          (管理快递单号)
    ↓ GET /api/plan                         (获取规划，自动触发快递查询)
Java Spring Boot
    ↓ 查未签收快递 → 组装 PlanRequest
    ↓ HTTP POST /generate-plan
Python FastAPI
    ↓ 调快递100 API 查询每条快递状态
    ↓ 融合进 LLM Prompt
    ↓ 返回 PlanResponse (含 parcel status)
Java Spring Boot
    ↓ 更新 parcel 表 (状态、is_delivered、last_checked)
```

### 快递100 API

#### 注册与额度

1. 前往 [快递100 API 开放平台](https://api.kuaidi100.com/) 注册账号
2. 在控制台获取 `customer`（客户ID）和 `key`（授权密钥）
3. 免费额度：每天 100 次查询，个人使用足够

#### 调用方式

- **端点**：`POST https://poll.kuaidi100.com/poll/query.do`
- **签名**：`MD5(param_json + key + customer)` 转大写
- **参数**：`customer`、`sign`、`param`（JSON: `{"com":"shunfeng","num":"SF1234567890"}`）
- **超时**：10s，超时或失败返回降级文本

#### 返回数据结构

```json
{
  "message": "ok",
  "state": "0",
  "status": "200",
  "data": [
    {
      "time": "2024-01-01 12:00:00",
      "ftime": "2024-01-01 12:00:00",
      "context": "快件已签收，签收人：本人",
      "location": "北京市朝阳区"
    }
  ]
}
```

#### 物流状态码

| 值 | 含义 | 处理 |
|---|---|---|
| 0 | 在途 | 保留，下次继续查 |
| 1 | 揽件 | 保留，下次继续查 |
| 2 | 疑难 | 保留，下次继续查 |
| 3 | 签收 | 标记 `is_delivered=true`，不再查 |
| 4 | 退签 | 保留 |
| 5 | 派件 | 保留，下次继续查 |
| 6 | 退回 | 保留 |

#### 快递公司编码映射

`parcel.py` 内置常用快递公司中文名 → 编码对照表，用户只需填"顺丰"即可自动转为 `shunfeng`。

支持的快递公司：顺丰、圆通、中通、申通、韵达、京东、EMS/邮政、极兔、德邦、百世、丹鸟、菜鸟、丰网、宅急送、优速、天天。

#### 环境变量

```bash
# agent_service/.env
KUAIDI100_CUSTOMER=your_customer_id
KUAIDI100_KEY=your_api_key
```

不配置则快递查询降级为"API 未配置"提示文本，不影响主流程。

### API 接口

#### 前端 → Java

**POST /api/parcel** — 添加快递单号
```json
→ Body: { "userId": 1, "trackingNo": "SF1234567890", "carrier": "顺丰", "remark": "耳机" }
← { "message": "快递单号添加成功", "id": 1 }
```

**GET /api/parcel?userId=1** — 查询快递列表
```json
← [{ "id": 1, "userId": 1, "trackingNo": "SF1234567890", "carrier": "顺丰", ... }, ...]
```

**DELETE /api/parcel/{id}** — 删除快递单号
```json
← { "message": "快递单号已删除" }
```

#### Java → Python（内部调用）

PlanRequest 新增字段：
```json
{
  "parcels": [
    { "tracking_no": "SF1234567890", "carrier": "顺丰", "remark": "耳机" }
  ]
}
```

PlanResponse 新增字段：
```json
{
  "parcels": [
    {
      "tracking_no": "SF1234567890",
      "carrier": "顺丰",
      "remark": "耳机",
      "state": "派件",
      "is_delivered": false,
      "latest_context": "快递员正在派送中",
      "latest_time": "2026-05-17 09:30:00"
    }
  ]
}
```

Java 端收到响应后，遍历 `parcels` 列表更新对应记录的 `status`、`last_checked`、`is_delivered`。

### 汇报效果示例

> 📦 快递物流
> - 你的耳机（顺丰 SF1234567890）今天到了，记得去取
> - 考研政治辅导书（中通 7512345678）还在路上，预计明天到

### 数据库表

```sql
CREATE TABLE parcel (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    tracking_no VARCHAR(100) NOT NULL,
    carrier VARCHAR(50) COMMENT '快递公司，如顺丰、京东',
    remark VARCHAR(100) COMMENT '备注，如"耳机"',
    status VARCHAR(200) COMMENT '最新物流状态',
    last_checked DATETIME,
    is_delivered BOOLEAN DEFAULT FALSE,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_delivered (user_id, is_delivered)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 第三部分：前端多页面改造

### 架构设计

```
main.js
├── createPinia()                         ← 全局状态
├── createRouter(createWebHashHistory)    ← Hash 路由
├── global.css                            ← 像素 CRT 样式
└── App.vue
    ├── <header> DayAgent 标题
    ├── <router-view />                   ← 页面视图
    │   ├── /login    → LoginView
    │   ├── /report   → ReportView
    │   ├── /summary  → SummaryView
    │   ├── /goals    → GoalsView
    │   ├── /parcel   → ParcelView
    │   └── /settings → SettingsView
    └── <BottomNav />                     ← 5 个导航入口

数据流（每个页面）：
  composable → api module → axios instance → Java/Python backend
       ↓
  view (编排容器) → sub-components (纯展示)
```

### UI 风格实现

#### CSS 架构

- **变量驱动**：所有颜色/字体/间距在 `:root` 中定义，12 个变量
- **通用 class**：`.crt-card`、`.pixel-btn`、`.pixel-input`、`.pixel-textarea`、`.pixel-select`、`.pixel-tag`
- **字体**：Press Start 2P（英文标题，Google Fonts CDN）+ zpix（中文正文，jsDelivr CDN）
- **CRT 效果**：`::after` 扫描线 + `::before` 暗角 vignette
- **动画**：skeleton-pulse、blink-cursor、fade-in-steps（全部 steps() 逐帧）
- **全局**：`*` 全局 reset，`image-rendering: pixelated`，`-webkit-font-smoothing: none`
- **滚动条**：像素化窄滚动条（6px）

#### 组件规范

- 每个 `.vue` 文件 `<script setup>` 不超过 60 行
- API 调用不放组件内，走 composable
- 四种状态全覆盖：loading（骨架屏）/ error（ErrorBlock）/ empty（EmptyState）/ data（正常展示）
- 底部导航用 CSS 像素字模拟图标（不需要图片资源）

### 发现并修复的 Bug

| # | 问题 | 修复 |
|---|---|---|
| 1 | **Result 包装器未解包** — Java `Result<T>` 统一包装，前端 `r.data` 拿到包装器，`.token` 为 `undefined` | `api/index.js` 响应拦截器自动识别 `code` 字段并提取 `data` |
| 2 | **Vite 代理端口错误** — `/courses` 代理指向 8006，Python 实际在 8000 | `vite.config.js` 端口改为 8000 |
| 3 | **FastAPI HTTPException detail 被丢弃** — 错误拦截器未匹配 `detail` 字段 | 在 `err.response.data` 链中添加 `detail` 检查 |
| 4 | **城市硬编码"北京"** — 前端写死覆盖用户 `.env` 的 `DEFAULT_LOCATION=南昌` | 移除前端硬编码，Java 默认值改为"南昌" |
| 5 | **Windows + uvicorn --reload 不兼容 Playwright** — SelectorEventLoop 导致 `NotImplementedError` | 去除 `--reload`，使用 `--loop asyncio` |
